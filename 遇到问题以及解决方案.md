# 一、RAG部分
## 1、数据清理
### 文档清洗
    - 将全角字符改成半角字符（跨平台兼容好）、删除连续的空行、清除html的相关标记（有的文档是由html转换过来的）、中文标点转换成英文标点（便于统一处理）。
    - 按正则表达式{第几条}进行分割，每条元数据保存法律来源和第几条的相关信息。
    - 对于token数量大于800的进行递归分割，保留100的over_lap。

## 2、问题
### 问题1：重启服务时，不想重新向量化（非常耗时），但是milvus无法读取全部数据(限制16384)，所以不能创建bm25检索器。如果只重新切分文档，但这样会导致文档的id属性与向量数据库的id不一致。
- 对切分的文档存储json文件用来回复bm25数据，存储id，法律来源，条目等元数据。
- 只要修改本地知识库，就需要重新向量化和保存的新的json文件。
- 如果后续增加删除和增加文件功能，可以创建一个法律文件与文档块的id映射表，根据id（向量库主键）从向量库中删除或增加对应的每个块，以及修改json文件。
- 目前一致性解决：只要删除增加文件，就需要重新建立向量库和json文件。

### 问题2：RAG检索——milvus[lite]不支持异步调用，刚开始定义的async检索是假异步
20个用户测试并发访问，RAG检索时间达到3秒多，使用loop.run_in_executor多线程（8个）+协程的方式，平均检索时间不到1秒钟，并发数多时，提高效果不大。




### 问题3：调用阿里百炼大模型，限流（主要瓶颈 RAG检索和agent节点）

## 3、评测与微调数据集构建
- 基于DISC公开数据集进行评测（question，reference（1~5条正样本），一个正样本算一条总共有60000多条）。
- 法律法规内容比较准确（每一条内容都是固定的，每个字都是按顺序的，每个字都是一样的，标点在解析文档的时候各个来源有所不同），可能在解析的时候标点存在差异，所以ROUGE-L相比于n-gram的ROUGE更合适。
- ROUGEL可能受标点换行符影响，评测数据集的标点与自己知识库的标点差异比较多，导致得分很低，去掉所有标点，只保留中英文和数字。
- 微调数据集构建：针对每一条样本，初始模型检索出来的ROUGEL值小于0.3且向量召回排序top3的作为负样本构造微调数据集，60000条数据。微调embedding模型。infoNCEloss，显存占用6GB左右，微调了不到2天。

# 二、智能体部分
## 1、上下文管理
### 上下文组成（模型上下文长度为m）
- 用户输入query（固定token数量 = m1）
- 固定系统提示词（固定token数量 = m2） 
- 检索的知识库内容（知识库最大块token数量 * 最大检索出的块数量 = m3）
- 聊天历史（摘要历史和最近几轮完整历史 = m4 = m - m1 - m2 - m3）

### 管理聊天历史
- 摘要历史token数量 = m4 * 0.2
- 完整历史token数量 = m4 * 0.8
- 采用redis缓存管理聊天记录，提高agent的响应速度，因为聊天记录需要频繁读取，填充到上下文中。
- 当添加新的聊天记录后，完整历史token数量超过阈值，则执行摘要压缩（队列先进先出压缩旧聊天记录），将压缩后的内容放入摘要队列中，如果添加新的摘要内容后超出摘要阈值，则执行摘要删除（队列先进先出删除旧摘要内容）。
- 采样arq + Redis 消息队列模式，执行添加聊天记录到数据库的操作。
- 本项目为什么选 Arq
    - 聊天记录非核心数据：即使少量丢失不影响核心业务
    - 聊天接口需低延迟：用户不希望等DB写入完才看到回复
    - 流量波动大：高峰期队列缓冲，减轻DB压力
## 2、React模式
单词检索可能无法检索出可以回答用户的内容，智能体需要根据返回的内容重构用户问题，得到更好的检索。
